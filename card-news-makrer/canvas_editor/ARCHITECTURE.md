## 1. Project Overview

The Canvas Editor MVP is a Next.js 14-based web application that enables operators to visually preview, edit, and approve card news specifications (`card_spec.json`) generated by upstream agents in the content production pipeline.

**Key Responsibilities:**
- Load and render card specs as 1080×1080 visual cards
- Enable inline text editing with character limits
- Support drag-and-drop card reordering
- Manage style adjustments (color, font size, layout, overlay opacity)
- Record all edits to Supabase for agent quality improvement
- Trigger SNS publishing workflow on approval

## 4. Core Data Structures

### CardSpec (from card_spec.schema.json)

```typescript
interface CardSpec {
  meta: {
    id: string;                    // "2026-03-09-001"
    topic: string;
    angle?: string;
    target_persona?: string;
    created_at: string;            // ISO 8601
    status: "draft" | "review" | "approved" | "published";
    sources?: string[];
  };
  cards: Card[];                   // 6-10 items
  sns: {
    instagram?: { caption: string; hashtags: string[] };
    threads?: { text: string };
  };
}

interface Card {
  index: number;                   // 1-based
  role: CardRole;
  text: {
    headline: string;              // max 15 chars
    body?: string;                 // max 50 chars
    sub_text?: string;
  };
  style: {
    layout: Layout;
    color_palette?: ColorPalette;
    font?: FontStyle;
  };
  background: {
    type: "image" | "gradient" | "solid";
    src?: string | null;
    prompt?: string | null;
    overlay_opacity?: number;      // 0-1
  };
}

type CardRole = "cover" | "empathy" | "cause" | "insight" | "solution" | "tip" | "closing" | "source" | "cta";
type Layout = "center" | "top-left" | "top-right" | "bottom-left" | "bottom-right" | "split";
```

### Supabase Tables

```sql
-- card_specs: Store complete card_spec.json
CREATE TABLE card_specs (
  id TEXT PRIMARY KEY,
  topic TEXT NOT NULL,
  status TEXT DEFAULT 'draft',
  spec JSONB NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- edit_logs: Track all manual edits
CREATE TABLE edit_logs (
  id BIGSERIAL PRIMARY KEY,
  spec_id TEXT REFERENCES card_specs(id) ON DELETE CASCADE,
  editor TEXT NOT NULL,
  field_path TEXT NOT NULL,       // e.g., "cards[2].text.headline"
  old_value TEXT,
  new_value TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- publish_reports: Track publishing events
CREATE TABLE publish_reports (
  id BIGSERIAL PRIMARY KEY,
  spec_id TEXT REFERENCES card_specs(id) ON DELETE CASCADE,
  platform TEXT NOT NULL,         // "instagram" | "threads"
  post_url TEXT,
  post_id TEXT,
  status TEXT DEFAULT 'pending',
  published_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

## 7. State Management Architecture (Zustand)

The `useCardStore` manages the entire card_spec state:

```typescript
interface CardState {
  // Data
  cardSpec: CardSpec | null;
  selectedCardIndex: number;
  unsavedChanges: boolean;
  autoSaveStatus: "idle" | "saving" | "saved" | "error";

  // Actions
  loadSpec: (spec: CardSpec) => void;
  selectCard: (index: number) => void;
  updateCardText: (cardIndex: number, field: string, value: string) => void;
  updateCardStyle: (cardIndex: number, styleUpdates: Partial<CardStyle>) => void;
  reorderCards: (fromIndex: number, toIndex: number) => void;
  updateSnsCaption: (platform: "instagram" | "threads", content: string) => void;
  setStatus: (status: CardStatus) => void;
  reset: () => void;

  // Internal
  markUnsaved: () => void;
  markSaved: () => void;
}
```

Features:
- **Auto-save**: Debounced (1 second) to Supabase after any change
- **Edit logging**: Every change triggers `editLog` recording
- **Undo support** (future): History snapshots can be stored
- **Type safety**: Full TypeScript with schema validation

## 9. API Routes

All endpoints require authentication (Supabase JWT token in header).

### Card Specs

```
GET    /api/card-specs              → list all specs
POST   /api/card-specs              → create new spec
GET    /api/card-specs/[id]         → fetch single spec
PUT    /api/card-specs/[id]         → update spec
DELETE /api/card-specs/[id]         → delete spec
POST   /api/card-specs/[id]/approve → approve + publish
POST   /api/card-specs/[id]/reject  → reject (set to draft)
```

### Edit Logs

```
POST   /api/edit-logs               → record an edit
```

### Health Check

```
GET    /api/health                  → {"status": "ok", "db": "connected"}
```

## 12. Database & RLS Policies

Supabase Row-Level Security (RLS) ensures data isolation:

```sql
-- card_specs: Users can see their own specs
CREATE POLICY "Users can view own specs"
  ON card_specs FOR SELECT
  TO authenticated
  USING (auth.uid()::text = owner_id);

CREATE POLICY "Users can update own specs"
  ON card_specs FOR UPDATE
  TO authenticated
  USING (auth.uid()::text = owner_id)
  WITH CHECK (auth.uid()::text = owner_id);

-- edit_logs: Users can insert logs for their specs
CREATE POLICY "Users can insert edit logs"
  ON edit_logs FOR INSERT
  TO authenticated
  WITH CHECK (
    spec_id IN (
      SELECT id FROM card_specs
      WHERE owner_id = auth.uid()::text
    )
  );
```

## 16. Security Considerations

1. **Authentication**: Supabase JWT (required for all API routes)
2. **Input Validation**: Zod schemas on client & server
3. **CORS**: Next.js built-in handling
4. **CSRF**: Built-in Next.js protection
5. **Content Security Policy**: Added via headers config
6. **Secrets**: Environment variables only, never hardcoded
7. **Data Privacy**: RLS policies enforce user isolation
